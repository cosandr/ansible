# CLAUDE.md

## Overview

Personal homelab infrastructure-as-code repo, driven entirely by Ansible. ~104 roles (mix of in-repo, git submodule, and Galaxy-installed), a static generated inventory, ansible-vault secrets backed by gopass, and Renovate for dependency version bumps.

## Never do this

- Never run Ansible playbooks or Ansible ad-hoc commands against any host in the inventory. Localhost is an exception, you may run against it for debugging Jinja expressions or other templating/looping logic.
- Never attempt to SSH into any host.
- Never run `ansible-vault`, `gopass` or otherwise attempt to bypass this restriction by using `ansible` ad-hoc commands.

## Commands

```bash
# install galaxy collections + roles listed in requirements.yml
ansible-galaxy install -r requirements.yml

# regenerate inventory after changing files/inventory/*.j2 templates or host list
make inventory

# lint
ansible-lint

# per-role molecule tests (only roles with a molecule/ dir; podman driver)
cd roles/<name> && molecule test

# Initialize new role (plain) - preferred, unless told otherwise
./new-role.sh <name>

# Initialize new role (submodule)
./new-role-submodule.sh <name>

# Remove submodule, use for submoduled roles
./rm-submodule.sh <path>
```

## Secrets

- ansible-vault is used throughout
- Convention: every `group_vars`/`host_vars` scope keeps secrets in a separate `vault.yml` (encrypted) alongside a plaintext `vars.yml` that references `vault_*`-prefixed variables. Don't merge them.
- The pre-commit hook (installed by `make git-hooks`, runs `vault-check.py`) blocks commits containing unencrypted content in files matched by the `ansible-vault` diff pattern in `.gitattributes` (`*vault.yml`, `*.key`, `*.csr`). Never bypass this by hand-editing around it.
- Wireguard keys are generated out-of-band (`wg genkey`/`wg genpsk`), not templated — stored directly in gopass under `network/<inventory_hostname>_wg_pk` / `_wg_psk`.

## Architecture

### Inventory generation

Inventory is a static INI file (`inventory/hosts`). It contains an autogenerated block (`# BEGIN/END AUTO GENERATED VARIABLES`) built by `playbooks/generate_inventory.yml`, which renders `files/inventory/{10,20,30,40}-net-*.j2` templates through custom lookup plugins (`generate_hosts`, `generate_network`) defined in the in-repo collection `ansible_collections/andrei/utils`. Group hierarchy (`servers` → `kvm_hv, libvirt, rom, rpi, webgw, nspawn`, etc.) is hand-maintained below the autogen block — editing it is safe, editing inside the autogen markers is not.

`group_vars/<group>/` and `host_vars/<host>/` are split into topic files (`firewall.yml`, `vars.yml`, `vault.yml`, ...) rather than one monolithic vars file per scope.

### Roles

Three sources, distinguishable by where they live:
- **In-repo** (majority, ~70+): plain custom roles under `roles/`.
- **Git submodules** (23, tracked in `.gitmodules`): the author's own forks of community roles (prometheus, grafana, nginx, wireguard, alertmanager, postgresql, ...).
- **Galaxy-installed** (gitignored, from `requirements.yml`). Install/update with `ansible-galaxy install -r requirements.yml`. If the role isn't pinned to a specific, `--force` must be passed to ansible-galaxy for it to be updated.

Role convention: vars prefixed with the role name (`node_exporter_*`), layout `defaults/main.yml`, `vars/main.yml`, `tasks/{main,install,configure}.yml`, `templates/*.service.j2`, `handlers/main.yml`. Despite names like `cadvisor`, `node_exporter`, `prometheus`, these deploy as **native systemd services**, not containers — actual Docker usage is limited to the `docker` host group (`playbooks/docker.yml`, via `geerlingguy.docker`).

Cross-role reuse: Common task snippets live in `tasks/common_servers.yml` and `tasks/configure_ssh_user.yml`, pulled in via relative `include_tasks` paths.

Playbooks are executable scripts (`#!/usr/bin/env -S ansible-playbook` shebang), named after their target group, often multi-play with tags scoping each play.

### Dependency updates

`.renovaterc.json5` drives version bumps via a custom regex manager that scans `group_vars`/`host_vars`/`defaults`/`vars` YAML for comment-tagged version pins:

```yaml
# renovate: datasource=github-releases depName=prometheus/node_exporter
node_exporter_version: "v1.12.0"
```

This produces the recurring "Update dependency X to vY (#NN)" commits.
